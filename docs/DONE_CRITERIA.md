# PromptCanvas Definition of Done

## Purpose

A feature is considered complete only when it satisfies engineering, design, testing, documentation, accessibility, and performance requirements.

Completion is based on quality rather than speed.

## Engineering Checklist

The feature:

- [ ] Compiles successfully
- [ ] Has no TypeScript errors
- [ ] Has no ESLint errors
- [ ] Has meaningful variable names
- [ ] Has reusable components
- [ ] Avoids duplicated code
- [ ] Follows project architecture

## UI Checklist

The feature:

- [ ] Matches the PromptCanvas design language
- [ ] Is responsive
- [ ] Works on desktop
- [ ] Works on tablet
- [ ] Works on mobile
- [ ] Uses consistent spacing
- [ ] Uses consistent typography

## Accessibility Checklist

The feature:

- [ ] Uses semantic HTML
- [ ] Supports keyboard navigation
- [ ] Has visible focus states
- [ ] Maintains sufficient color contrast
- [ ] Respects reduced-motion preferences

## Performance Checklist

The feature:

- [ ] Avoids unnecessary re-renders
- [ ] Loads efficiently
- [ ] Uses optimized assets
- [ ] Keeps animations smooth
- [ ] Avoids unnecessary client-side JavaScript

## Documentation Checklist

Documentation is updated where appropriate:

- [ ] `CHANGELOG.md`
- [ ] `PROJECT_STATUS.md`
- [ ] Relevant architecture documents

## Testing Checklist

Verify:

- [ ] Build succeeds
- [ ] Manual testing completed
- [ ] Console contains no errors
- [ ] Route navigation works correctly
- [ ] Forms validate correctly
- [ ] Error states are handled

## Git Checklist

Before every commit:

- [ ] Working tree is clean
- [ ] Commit message follows Conventional Commits
- [ ] Project builds successfully

## Final Question

Before marking a feature as complete, ask:

_"Would I confidently demo this feature to a recruiter, a client, or a teammate?"_

If the answer is no, continue improving it.

---

A feature is complete only when it is reliable, maintainable, understandable, and ready for production.
